# Montree — Full App Audit
**Date:** 2026-06-10 · **Scope:** connection integrity, auth/security, integrations/env, product function vs. purpose
**Method:** four parallel code audits over `whale/` (Glob/Grep/Read), top findings re-verified by hand. Bash sandbox was down, so everything here is from static reading of source — no runtime probes.

---

## How to read this
The app's **core Montree surface (teacher/parent/admin/agent) is well-built and tightly wired.** The serious problems cluster in three places: (1) a leaked production credential, (2) a handful of unauthenticated write endpoints, and (3) a large legacy `app/admin/*` + `app/teacher/*` tree and a games hub that are still reachable but point at deleted or nonexistent APIs. Fix the credential today; the rest is triage.

Counts: ~509 API route files (758 handlers); ~770 client fetch sites + 182 `montreeApi()` calls checked; 397 `montree/**` routes scanned for auth; full env-var inventory built.

Reachable from the teacher Tools page. It lies to the user; this is the worst trust-breaker in the product. Hide it until real.

**6. ~25 routes bypass the hardened Supabase client.** `lib/supabase-client.ts` has the 12s timeout + retry that fixed the zombie-socket hangs, but many routes build a raw `createClient` without it — e.g. `montree-home/*`, `reports/{send,photos,available-photos,unreported}`, `progress/{route,update,batch-master}`, `weekly-assignments`. On a stale Railway↔Supabase socket these hang indefinitely — the exact bug you already fixed elsewhere. Sweep raw `createClient` → `getSupabase()`.

**7. Module-scope service client + no auth on three routes.** `app/api/classroom/[classroomId]/curriculum/route.ts:5` builds a service-role client at import time with `!` (missing env crashes the server at boot) **and** the GET has zero auth — read any classroom's curriculum by UUID. Same pattern: `students/[studentId]/quick-place/route.ts:6`, `onboard/principal/route.ts:7`.

**8. Cross-tenant write gap in `children` POST.** `app/api/montree/children/route.ts:32` authenticates but never checks `classroom.school_id === auth.schoolId` — a teacher in School A can create a child in School B's classroom by supplying its ID. (The GET path scopes correctly; the POST doesn't.)

**9. Super-admin password in URL query strings.** `super-admin/{impact-fund,npo-applications,reduced-rate-applications}/route.ts` accept `?password=` on GET. Query strings are logged by Cloudflare + Railway (and `verify-request.ts` logs pathnames). The master password leaks into logs. Also `impact-fund` uses non-timing-safe `!==` to gate a financial ledger write. Move to header/body + timing-safe compare.

**10. Expensive AI is unmetered.** Astra's Tracy routes (`admin/tracy/draft-response/route.ts:186`, `scan-thread/route.ts:154`) call Opus with **no `logUsage`** and no price row in `api-usage.ts` (unknown models silently bill at Sonnet rates). `admin/principal-agent/route.ts` skips `checkAiBudget` entirely (the guru route enforces it). A chatty principal blows past the school AI budget cap, invisibly to your P&L. Add pricing rows + the budget gate.

**11. Voice/agent fetches have no timeout.** `lib/montree/tracy/tool-executor.ts:154` `internalGet` (the most-used Astra tools) lacks the AbortController its own comment promises; `lib/montree/voice-agent/agent.ts:160,183` Agora join/leave are bare fetches — a failed "leave" orphans a voice agent burning Agora minutes + tokens until the 60s idle timer (which won't fire mid-conversation).

**12. One server-to-self hairpin remains.** `app/api/weekly-planning/upload/route.ts:125` fetches `/api/admin/curriculum/sync-all` through `request.url` (public origin) instead of `127.0.0.1:$PORT` — the footgun you fixed in `tool-executor.ts`. Auto-sync silently no-ops on Railway region moves.

**Offline durability is photos-only.** `lib/montree/offline/types.ts` queues `PhotoQueueEntry` only. With the known ~20% origin drop, a teacher tagging a work / writing an observation mid-classroom loses the write with no retry — the pedagogical data attached to durable photos is itself not durable. Extend the queue to progress/observation writes.

**Two service workers coexist.** `next-pwa` (`next.config.ts:205`) runs alongside the hand-rolled `public/montree-sw.js` (v10), with overlapping `_next/static` cache rules — the stale-shell class of bug the v7→v10 history chronicles. Pick one worker.

**Other notable:** YouTube API key shipped to browser (`lib/youtube/search.ts:14`) — restrict by referrer or proxy it. `resolveReportModel` fails *closed* (`resolve-model.ts:66`) — a transient DB error silently strips AI from a paying school. Guru freemium counter fails *open* (`guru/route.ts:400`) — DB error resets the daily paywall to 0. ## Product / concept assessment vs. purpose
Montree's job is: let Montessori teachers capture and track child progress, give principals oversight, and keep parents engaged. Against that, the build is genuinely strong on capture→tag→report, but has structural gaps:

**1. No Thai.** i18n supports 11 languages (en/zh/es/de/fr/pt/nl/it/ja/ko/uk/ru) — none of them Thai, despite deployment at a Thailand school, and the parent dashboard hardcodes EN/ZH labels. This is the single biggest market miss. The i18n machinery is solid; adding `th` is mostly mechanical (plus a Thai output mode in report prompts).

**2. The teacher review loop has no SLA.** Capture is great, but "tag the photos later" lives in a 3,669-line photo-audit page a teacher must remember to visit. There's no "14 untagged photos from today" chip on the dashboard. The data (`identification_status`) already exists — surfacing a count closes the loop cheaply. Highest value-per-effort retention fix.

**3. Analytics are child-centric, not curriculum-centric.** Nobody can answer the two questions a directress actually plans around: "which works/areas has *no one* touched this term?" and "which children haven't been observed in 2 weeks?" The proactive version was built for principals (`SHOW_ASTRA_PROACTIVE`) then turned off; the teacher-facing one was never built. Highest-value analytics gap.

**4. Parent engagement is pull-only.** Rich parent dashboard, but nothing brings parents back — no push (the SW push handler and `web-push` dep exist but no Montree surface ever subscribes), no email digest, report sending is teacher-initiated. Engagement decays to whenever a teacher manually sends. A weekly cron digest + the already-written push handler is most of the build, and your cron infra already exists.

**5. The data model assumes one classroom per teacher.** `auth.ts` session carries `classroom | null` (singular). A floating ESL/specialist teacher — the actual staffing pattern — can't exist. And soft-deleting a classroom still hides its *active* children from every roster (`admin/students/route.ts:19`, ~25 routes share the join) with no "archived" surfacing. Both are real operational traps.

**6. Voice Astra is built but hard-off.** `SHOW_VOICE_ASTRA = false` in code (not a flag), and school-level flags like `voice_astra` have no UI to toggle (the Features page is per-classroom and bails for principals). Significant shipped capability that's unreachable without a redeploy.

**7. Engineering hygiene is the risk multiplier.** One test file in the whole repo (`tests/auth-tokens.test.ts`), `ignoreBuildErrors: true` over ~5,233 tsc errors, no `error.tsx` boundaries outside the teacher dashboard (a render error gives principals/parents the Next.js crash screen), 23+ `console.log` in prod pages. The `audit:ts-budget` script exists but isn't wired to CI. None of this is user-visible today; all of it makes every future change riskier.
